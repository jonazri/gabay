### IPC Handler Registry (Phase 2)

Created `.claude/skills/ipc-handler-registry/` with:
- `add/src/ipc-handlers.ts` — Map-based handler registry (26 lines)
- `add/src/ipc-handlers.test.ts` — 3 tests (register/retrieve, undefined for unknown, throws on duplicate)

**Architectural deviation from plan:** Handler dispatch was placed BEFORE the switch statement (not in the default case). This is better — avoids modifying the switch block, and registered handlers take precedence over upstream cases.

## Issues to Fix Before Phase 4

### 1. whatsapp-replies `_accumulated` overlays carry pruned dead code

**Files:** `whatsapp-replies/modify/src/db.ts` and `whatsapp-replies/modify/src/db.test.ts`

These still contain the 4 pruned reaction query functions (`getReactionsForMessage`, etc.) and their tests (~300 lines). Since whatsapp-replies uses `modify_base: _accumulated` for these files, the overlays need rebuilding against the new accumulated state (post add-reactions pruning). This is already tracked as Task 4.3 (rebuild all _accumulated overlays) but should be done before the upstream merge to avoid carrying dead code forward.

### 2. `storeMessageDirect()` persists in 2 other skill overlays

**Files:** `add-reactions/modify/src/db.ts` (line 291), `add-group-lifecycle/modify/src/db.ts` (line 268)

Zero callers anywhere in the codebase. Should be removed when these overlays are next rebuilt (during Phase 4).

### 3. `scripts/migrate-reactions.ts` still in working tree

Untracked file at repo root. The skill file was deleted and manifest updated, but this working tree copy remains. Just delete it: `rm scripts/migrate-reactions.ts`

### 4. ipc-handlers.test.ts singleton reset fragility

The test `beforeEach` reimports the module via `await import('./ipc-handlers.js')` but doesn't call `vi.resetModules()`. The handler Map may carry state between tests. Currently works because each test registers a unique key (`test_type`, `nonexistent`, `dup`). Add `vi.resetModules()` to make it robust.

### 5. notify.ts uses `MAIN_GROUP_FOLDER` pattern

The shared utility matches `g.folder === folder` using `MAIN_GROUP_FOLDER`. Upstream v1.2.1 replaces this with `group.isMain === true`. Must update during Phase 4 pattern migration. No impact today since no callers use it yet.
