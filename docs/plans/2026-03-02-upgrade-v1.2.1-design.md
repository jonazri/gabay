# Upgrade to NanoClaw v1.2.1 — Design

## Problem

Upstream NanoClaw v1.2.1 introduces a multi-channel architecture refactor (#500). Our fork has 15 installed skills, 6 of which ship full-file copies of `src/ipc.ts` (~400 lines each) and `src/index.ts` (~500+ lines each) to contribute 10-50 lines of actual change. This duplication makes upstream merges painful: the three-way merge algorithm can't handle context shifts when multiple skills modify nearby regions of the same large file.

A previous merge attempt (this session) confirmed that textual three-way merge rebasing fails at scale for these overlapping full-file copies.

## Approach: Refactor Then Merge

Simplify skills FIRST on the old upstream, THEN merge. Skills become minimal registrations instead of full-file copies. The subsequent merge has minimal conflict surface.

## Component 1: IPC Handler Registry

### Current State
`src/ipc.ts` has a monolithic switch statement with 6 upstream cases. Six fork skills each ship a full copy of this 400-line file to add one or two `case` blocks.

### Design
New file: `src/ipc-handlers.ts`

```typescript
type IpcHandler = (
  data: Record<string, any>,
  deps: IpcDeps,
  context: IpcContext,
) => void | Promise<void>;

interface IpcContext {
  sourceGroup: string;
  isMain: boolean;
  folderIsMain: Map<string, boolean>;
}

const handlers = new Map<string, IpcHandler>();

export function registerIpcHandler(type: string, handler: IpcHandler): void {
  handlers.set(type, handler);
}

export function getIpcHandler(type: string): IpcHandler | undefined {
  return handlers.get(type);
}
```

In `src/ipc.ts`, the switch `default:` case delegates to the registry:

```typescript
default: {
  const handler = getIpcHandler(data.type);
  if (handler) {
    await handler(data, deps, { sourceGroup, isMain, folderIsMain });
  } else {
    logger.warn({ type: data.type }, 'Unknown IPC task type');
  }
}
```

### Skill Migration
Each IPC skill replaces its `modify/src/ipc.ts` (full file) with `add/src/ipc-handlers/<name>.ts` (handler registration, ~20 lines). Skills affected:
- add-reactions → registers `reaction` handler
- add-refresh-oauth → registers `refresh_oauth` handler
- add-group-lifecycle → registers `unregister_group` handler
- add-google-home → registers `google_assistant_command`, `reset_google_conversation`, `google_assistant_health`
- add-shabbat-mode → registers Shabbat guard (middleware or pre-check)
- whatsapp-replies → no IPC handler, but benefits from simplified ipc.ts overlay removal

## Component 3: YAGNI Pruning

From the skills audit (2026-03-02):
- **add-reactions**: Remove 4 unused DB query functions (~80 lines): `getReactionsForMessage`, `getReactionsByUser`, `getMessagesByReaction`, `getReactionStats`. Remove `migrate-reactions.ts` (redundant with `createSchema()`).
- **add-container-hardening**: Remove stale `OPENAI_API_KEY` from manifest `container_secrets`.
- **add-perplexity-research**: Fix `depends` list (remove google-home, container-hardening, whatsapp-search).
- **whatsapp-replies**: Remove `storeMessageDirect()` (no callers).
- **add-voice-transcription-elevenlabs**: Consider gating metadata enrichment (~80 lines of sentiment/pace analysis). Use `logger` instead of `console.warn/error`.

## Component 4: Index.ts Simplification

With IPC handlers extracted and shared utilities available, index.ts overlays shrink. Remaining modify patterns:
- **add-reactions**: StatusTracker integration, emoji lifecycle callbacks, cursorBeforePipe rollback (~100 lines after extracting IPC callback wiring)
- **add-refresh-oauth**: Token refresh scheduler start/stop, auth error retry in runAgent (~30 lines after extracting notifyMainGroup and auth-utils)
- **add-group-lifecycle**: `deleteRegisteredGroup` import + `unregisterGroup` function (~11 lines)
- **add-google-home**: `shutdownGoogleAssistant` import + shutdown call (~2 lines)
- **add-shabbat-mode**: Shabbat guard + candle lighting notifier + schedule init (~50 unique lines, uses `_accumulated`)
- **whatsapp-replies**: `quotedKey?` parameter on sendMessage (~5 unique lines, uses `_accumulated`)

Goal: no modify file for index.ts exceeds ~200 lines (currently some are 800+).

## Execution Sequence

### Phase 1: Reset & Prepare
1. Clean pre-merge state (done: `git reset --hard pre-update-44ed0b6-20260302-224159`)
2. Create branch: `feat/upgrade-v1.2.1`

### Phase 2: IPC Handler Registry
3. Create `src/ipc-handlers.ts` with handler registry
4. Modify `src/ipc.ts` — add registry delegation in `default` case
5. For each of 6 IPC skills: convert to handler registration, remove `modify/src/ipc.ts`
6. Verify: `npm run apply-skills && npm run build && npm test`

### Phase 3: Shared Utilities & Cleanup

### Phase 4: Upstream Merge
12. `git fetch upstream && git merge upstream/main`
13. Resolve remaining conflicts (should be minimal)
14. Update patterns: `MAIN_GROUP_FOLDER` → `group.isMain`, `WhatsAppChannel` → channel factory, etc.
15. `npm run build && npm test`

### Phase 5: Validation
16. Full test suite (451 tests)
17. Manual review of diff vs upstream
18. Verify service starts

## Success Criteria

- All 15 skills apply cleanly with `npm run apply-skills`
- `npm run build` succeeds
- `npm test` passes (all existing tests)
- No skill modify file exceeds ~200 lines
- Zero functional regressions
- Clean git history (each phase is a logical commit)

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| IPC registry conflicts with upstream's ipc.ts changes | Registry is additive (new file + small default case). Upstream switch cases untouched. |
| Shared utilities create new merge points | `notify.ts` and `auth-utils.ts` are new files, not modifying upstream. |
| Functional regressions from YAGNI pruning | Only remove zero-caller code. Full test suite after each change. |
| `_accumulated` rebuild still needed | Shorter chain with fewer overlapping skills. Simpler to rebuild. |
