## Merge Blockers

Issues that should be addressed before merging this PR.

### Must-fix (resolved)

Two issues found by Qodo review (2026-03-04), both in upstream migration code:

1. **Main flag lost on JSON‚ÜíSQLite migration** (`39d07d1`) ‚Äî `is_main` backfill ran before `migrateJsonState()`, so `INSERT OR REPLACE` overwrote `is_main=1` with `0` for groups migrated from `registered_groups.json`. Fixed: set `group.isMain = true` for `folder='main'` during migration loop.

2. **Register step breaks old DB** (`4c56a31`) ‚Äî `setup/register.ts` INSERT included `is_main` column but `CREATE TABLE IF NOT EXISTS` doesn't add columns to existing tables. Fixed: added `ALTER TABLE` migration before INSERT.

The remaining issues below are pre-existing technical debt that this upgrade did not introduce ‚Äî they exist in the `_accumulated` overlays which were rebuilt from a state that already had these patterns.

### Should-fix (pre-existing debt surfaced by this upgrade)

1. **`add-shabbat-mode` and `whatsapp-replies` index.ts use `new WhatsAppChannel` instead of channel factory loop.** These `_accumulated` overlays were captured from a state that uses direct WhatsApp instantiation. Works because this fork always has WhatsApp installed, but diverges from upstream's multi-channel pattern. Would break if another channel skill were installed without WhatsApp.

2. **`notify.ts` is dead code.** Created with tests but never imported by production code. Three index.ts overlays carry inline `notifyMainGroup()` closures with different signatures (closure-based vs explicit parameters). Either wire callers to use it, or remove it to avoid confusion.

3. **Candle lighting notifier bypasses `findChannel`.** Line 845 of applied index.ts calls `whatsapp.sendMessage` directly instead of going through the channel abstraction. Would fail if the main group were registered via a non-WhatsApp channel.

4. **2 stale ipc.ts overlays.** `add-shabbat-mode/modify/src/ipc.ts` and `add-reactions/modify/src/ipc.ts` are pre-registry copies ‚Äî they're missing the `getIpcHandler` import and handler registry `default` case delegation. No runtime impact (overridden by whatsapp-replies, which is last in install order), but confusing for anyone reading the skill source.

5. **whatsapp-replies carries 1,332 lines of unnecessary overlays.** Its `modify/src/index.ts` (870 lines) and `modify/src/ipc.ts` (462 lines) contribute zero unique content ‚Äî they're identical to the accumulated state from prior skills.

6. **Intent doc references stale pattern.** `add-task-scheduler-fixes/modify/src/task-scheduler.ts.intent.md` still references `MAIN_GROUP_FOLDER`.

## Phase 5: Simplification (future work)

The IPC handler registry proved that a registration pattern can eliminate large overlays. The same approach can be applied to `index.ts`, where 6 skills ship 558‚Äì869 line copies to contribute 0‚Äì165 unique lines each.

### P1 ‚Äî Do in the next PR (low risk, high ROI)

#### 5.1: Remove unnecessary whatsapp-replies overlays

**Lines saved:** 1,332
**Complexity:** Low
**What:** Delete `whatsapp-replies/modify/src/index.ts` and `modify/src/ipc.ts`. These contribute zero unique content ‚Äî the build system will use the accumulated state from prior skills, which is identical. Also fixes the `WhatsAppChannel` direct-instantiation divergence in the whatsapp-replies overlay.

#### 5.2: Lifecycle hook system

**Lines saved:** ~1,328
**Complexity:** Medium

Create a `LifecycleRegistry` (same pattern as IPC handler registry) with hooks:
- `onStartup(fn)` ‚Äî called during `main()` after DB init, before channels connect
- `onShutdown(fn)` ‚Äî called during graceful shutdown
- `onChannelsReady(fn)` ‚Äî called after all channels are connected
- `shouldProcessMessages(fn)` ‚Äî predicate that can block message processing
- `onPostSilence(fn)` ‚Äî called when `shouldProcessMessages` transitions from false to true

Skills register hooks via barrel imports (like IPC handlers do today).

**Impact per skill:**
- add-google-home: index.ts overlay **eliminated** (4 unique lines ‚Üí `onShutdown` hook registration, no overlay needed)
- add-group-lifecycle: index.ts overlay **eliminated** (13 unique lines ‚Üí `unregisterGroup` passed through IPC handler deps, no overlay needed)
- add-shabbat-mode: index.ts overlay **reduced from 869 to ~100 lines** (registers `shouldProcessMessages` + `onPostSilence` + `onStartup` + `onShutdown` hooks; no longer needs `_accumulated` base)

#### 5.3: Wire notify.ts callers or remove it

**Lines saved:** 0 (cleanup)
**Complexity:** Low

Either refactor the 3 inline `notifyMainGroup()` closures to use the shared `notify.ts` (requires adapting signatures), or delete `notify.ts` and its test to eliminate the dead code. If the lifecycle hook system (5.2) lands, `notifyMainGroup` could be called from hook registrations rather than inline closures, making the shared version more natural to use.

#### 5.4: Clean up stale ipc.ts overlays

**Lines saved:** 0 (cleanup)
**Complexity:** Low

The `add-shabbat-mode/modify/src/ipc.ts` and `add-reactions/modify/src/ipc.ts` overlays are pre-registry copies missing the handler dispatch. They're overridden at runtime but confusing. ### P2 ‚Äî Next upgrade cycle (higher complexity)

#### 5.5: Message lifecycle events for reactions

**Lines saved:** ~640
**Complexity:** High

Create an event emitter or callback registry for the message processing pipeline (`onMessageReceived`, `onAgentStarting`, `onAgentFirstOutput`, `onAgentSuccess`, `onAgentError`). #### 5.6: Extend IPC handler registry to message dispatch

**Lines saved:** ~388
**Complexity:** Low-Medium

The IPC handler registry currently only covers `processTaskIpc` (the `tasks/` directory). Message dispatch (`messages/` directory) handles `message` and `reaction` types inline. Extending the registry to cover message types would let the reactions skill register a `reaction` message handler instead of modifying `ipc.ts`. Same authorization pattern, natural extension of the existing registry.
