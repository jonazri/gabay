## Merge Blockers

Issues that should be addressed before merging this PR.

### Must-fix

None. The upgrade is functional: all 17 skills apply, 438 tests pass, 0 type errors, 0 regressions. The issues below are pre-existing technical debt that this upgrade did not introduce тАФ they exist in the `_accumulated` overlays which were rebuilt from a state that already had these patterns.

### Should-fix (pre-existing debt surfaced by this upgrade)

1. **`add-shabbat-mode` and `whatsapp-replies` index.ts use `new WhatsAppChannel` instead of channel factory loop.** These `_accumulated` overlays were captured from a state that uses direct WhatsApp instantiation. Works because this fork always has WhatsApp installed, but diverges from upstream's multi-channel pattern. Would break if another channel skill were installed without WhatsApp.

2. **`notify.ts` is dead code.** Created with tests but never imported by production code. Three index.ts overlays carry inline `notifyMainGroup()` closures with different signatures (closure-based vs explicit parameters). Either wire callers to use it, or remove it to avoid confusion.

3. **Candle lighting notifier bypasses `findChannel`.** Line 845 of applied index.ts calls `whatsapp.sendMessage` directly instead of going through the channel abstraction. Would fail if the main group were registered via a non-WhatsApp channel.

4. **2 stale ipc.ts overlays.** `add-shabbat-mode/modify/src/ipc.ts` and `add-reactions/modify/src/ipc.ts` are pre-registry copies тАФ they're missing the `getIpcHandler` import and handler registry `default` case delegation. No runtime impact (overridden by whatsapp-replies, which is last in install order), but confusing for anyone reading the skill source.

5. **whatsapp-replies carries 1,332 lines of unnecessary overlays.** Its `modify/src/index.ts` (870 lines) and `modify/src/ipc.ts` (462 lines) contribute zero unique content тАФ they're identical to the accumulated state from prior skills.

6. **Intent doc references stale pattern.** `add-task-scheduler-fixes/modify/src/task-scheduler.ts.intent.md` still references `MAIN_GROUP_FOLDER`.

## Phase 5: Simplification (future work)

The IPC handler registry proved that a registration pattern can eliminate large overlays. The same approach can be applied to `index.ts`, where 6 skills ship 558тАУ869 line copies to contribute 0тАУ165 unique lines each.

### P1 тАФ Do in the next PR (low risk, high ROI)

#### 5.1: Remove unnecessary whatsapp-replies overlays

**Lines saved:** 1,332
**Complexity:** Low
**What:** Delete `whatsapp-replies/modify/src/index.ts` and `modify/src/ipc.ts`. These contribute zero unique content тАФ the build system will use the accumulated state from prior skills, which is identical. Also fixes the `WhatsAppChannel` direct-instantiation divergence in the whatsapp-replies overlay.

#### 5.2: Lifecycle hook system

**Lines saved:** ~1,328
**Complexity:** Medium

Create a `LifecycleRegistry` (same pattern as IPC handler registry) with hooks:
- `onStartup(fn)` тАФ called during `main()` after DB init, before channels connect
- `onShutdown(fn)` тАФ called during graceful shutdown
- `onChannelsReady(fn)` тАФ called after all channels are connected
- `shouldProcessMessages(fn)` тАФ predicate that can block message processing
- `onPostSilence(fn)` тАФ called when `shouldProcessMessages` transitions from false to true

Skills register hooks via barrel imports (like IPC handlers do today).

**Impact per skill:**
- add-google-home: index.ts overlay **eliminated** (4 unique lines тЖТ `onShutdown` hook registration, no overlay needed)
- add-group-lifecycle: index.ts overlay **eliminated** (13 unique lines тЖТ `unregisterGroup` passed through IPC handler deps, no overlay needed)
- add-shabbat-mode: index.ts overlay **reduced from 869 to ~100 lines** (registers `shouldProcessMessages` + `onPostSilence` + `onStartup` + `onShutdown` hooks; no longer needs `_accumulated` base)

#### 5.3: Wire notify.ts callers or remove it

**Lines saved:** 0 (cleanup)
**Complexity:** Low

Either refactor the 3 inline `notifyMainGroup()` closures to use the shared `notify.ts` (requires adapting signatures), or delete `notify.ts` and its test to eliminate the dead code. If the lifecycle hook system (5.2) lands, `notifyMainGroup` could be called from hook registrations rather than inline closures, making the shared version more natural to use.

#### 5.4: Clean up stale ipc.ts overlays
